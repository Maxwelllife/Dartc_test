Інструкція для локального запуску:

  1. Перейти в папку проєкту:

  cd /Dartc_test

  2. Встановити залежності:

  npm install

  3. Запустити локальний dev-сервер:

  npm run dev

  4. Відкрити адресу, яку покаже Vite в терміналі.

  http://localhost:5173

  Якщо треба, щоб сайт був доступний по мережі:

  npm run watch

  Для перегляду production-збірки:

  npm run build

 Збирав:
  за допомогою Vite.

  Інструменти, бібліотеки та підходи:

  - Vite для локальної розробки та production build
  - Sass/SCSS для стилів
  - Splide для hero-слайдера він найлегший для фронтових сторінок
  - modern-normalize для базового нормалізування стилів
  - нативний JavaScript (ES modules) без фреймворків
  - підхід до стилів: наближений до BEM
  - адаптивна верстка через mobile-first і SCSS mixins для брейкпоінтів
  - простий локальний mock submit для форми без реального бекенда
  - FSD архітектура

  Що було налаштовано у проєкті:

  - багатосторінкова збірка через Vite rollupOptions.input
  - окремі сторінки: головна, категорії, сторінка товару, checkout
  - підключення шрифтів Figtree
  - структура SCSS по блоках/віджетах
  - мобільне меню з dropdown-пунктами
  - hero-слайдер
  - форма з валідацією і mock-відправкою
  - панель favorites/cart з локальним станом через localStorage
  - SEO-базові мета-теги: title, description, keywords для всіх сторінок


Проблеми макета. Які рішення були прийняті і чому були додані додаткові сторінки.

Макет має серйозну проблему у візуаліт та в логіці продукту це видно по структурі сторінки. 
По макету неочевидно, яку саме сторінку ми бачимо: це може виглядати як home, як product page, як category page або як промо-лендінг одного товару. 
На одному екрані одночасно змішані всі фічі з різних сторінок таке відчуття, що свторено завдяки поганому промту для ШІ. 
А так як ставити запитання було нікому, а я намагався, то довелося приймати рішення самостійно.
Саме тому в реалізації з’явилося кілька окремих сторінок-заглушок - це була спроба закрити логічні дірки в макеті і зробити навігацію цілісною. Інакше частина кнопок, пунктів меню та користувацьких сценаріїв
виглядала б нефункціонально або суперечила б очікуванням користувача.
Зазвичай, якщо фіча не закінчина або не працює її не варто і показувати, але за такою логікою тут тоді нічого показути. Тому довелося додатково проектувати поведінку інтерфейсу, а не лише відтворювати макет.
